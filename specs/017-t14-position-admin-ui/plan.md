# Implementation Plan: Position Template admin UI (T14)

**Branch**: `017-t14-position-admin-ui` (stacked on `016-rubric-read-endpoint`) | **Date**: 2026-06-24 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/017-t14-position-admin-ui/spec.md`

## Summary

The recruiter UI to manage Position Templates — a list page (`/positions`) and a
create/edit form (`/positions/new`, `/positions/[id]`) — built on the committed
`openapi.yaml` (`/position-templates` CRUD + `/rubric/active`). Establishes the
frontend's first data layer (generated OpenAPI client + React Query) and its
first feature screens, following the design system (tokens, shadcn primitives,
Ukrainian labels, WCAG AA). Screen spec: `docs/design/screens/16-recruiter-positions/`.

## Technical Context

**Language/Version**: TypeScript (strict), Next.js 15 App Router, React 18
**Primary Dependencies (new)**: `@tanstack/react-query` (server state), `openapi-fetch` (typed client), `openapi-typescript` (dev — type gen), `msw` (dev — test HTTP mocking). New shadcn primitives: **select, checkbox, textarea**.
**Storage**: none (consumes the backend API)
**Testing**: jest + `@testing-library/react` + `user-event` + MSW (jsdom), in the `docker-compose.test.yml` `frontend` service
**Target Platform**: light-theme web (laptop; tablet best-effort), §14 design principle
**Constraints**: visual discipline (tokens only, no hex / arbitrary brackets / `dark:`), WCAG 2.2 AA, Ukrainian-first labels (§11), generated client (no hand-written types)
**Scale/Scope**: 2 routes + a form component + a table; ~3 new deps + 3 shadcn primitives; generated client + React Query hooks; jest tests

**Structure Decision**: First data layer = generated `openapi-fetch` client typed
from `openapi.yaml` (committed `schema.d.ts`) + thin React Query hooks in
`src/api/`. Pages stay thin; the table and form are feature components under
`components/positions/`. The QueryClient provider mounts in `layout.tsx` (the
convention's designated place for providers).
