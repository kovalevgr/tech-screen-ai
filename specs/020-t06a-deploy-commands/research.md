# Research — T06a deploy commands (`/deploy` + `/promote` + `/rollback`)

**Date**: 2026-07-05 · **Feature**: [spec.md](./spec.md) · Decisions D1–D12. Items marked **[repo-verified]** were checked against the tree at `main` (post-PR #18); nothing here was verified against live GCP — this task runs with **zero cloud mutations** (author-only Terraform, no gcloud calls), so live checks belong to the operator sweep in [quickstart.md](./quickstart.md).

## D1. Command surface: three `workflow_dispatch` workflows, invoked via `gh workflow run`

- **Decision**: `/deploy`, `/promote N`, `/rollback` are `.github/workflows/{deploy,promote,rollback}.yml`, each `workflow_dispatch` with typed inputs (`env`, `service` everywhere; `git_ref` on deploy; `percent` on promote; optional `revision` on rollback). The slash-command names remain the playbook verbs; the invocation is `gh workflow run <file> -f …`.
- **Rationale**: the implementation-plan contract names exactly these three files; `workflow_dispatch` gives typed/enum inputs (choice inputs cannot carry injection payloads), an audit trail (who dispatched what, when, with which inputs — GitHub retains it), and WIF compatibility. GitHub-side access control (who can dispatch) rides on repo write access at MVP — the `techscreen-deployers` group maps onto that until SSO-based deploy roles exist.
- **Alternatives considered**: **ChatOps trigger** (`issue_comment` parsing `/deploy`) — rejected for MVP: parses untrusted comment text (exactly the injection surface FR-013 bans) and adds a state machine for little gain; **ADR-012's "deploy-techscreen Claude Code skill"** — a skill can *call* `gh workflow run`, but the executable truth must live in CI where WIF exists; the ADR's mechanism sentence is superseded by these workflows (release philosophy unchanged). **Reusable composite action** — premature for three small workflows.

## D2. Migrations: operator-run, not CI-run *(mandated research question)*

- **Decision**: `/deploy` does **not** run `alembic upgrade head`. Schema application stays an operator runbook step (Auth Proxy + `techscreen_migrator`), exactly as executed for T06 (specs/018 quickstart §6). `/deploy` enforces §10 *approval* (the label gate, D7) — not §10 *application*.
- **Rationale**, honestly argued:
  1. **Credential shape (§5/§6)**: the migrator role is passwordless-at-rest — the standing pattern (specs/018 quickstart §4, password-hygiene note) is *rotate-on-demand*: the operator sets a fresh password per migration session and discards it. CI applying DDL would need a standing migrator credential (a GitHub secret or a Secret Manager entry readable by a CI SA) — precisely the long-lived credential §5/§6 spends its entire budget avoiding. An IAM-DB-auth CI user (the flag-sync pattern) *could* avoid the password, but would need `GRANT`s approaching table ownership for DDL — a CI identity that can `ALTER TABLE turn_trace` is a §3 hazard parked in every workflow run.
  2. **§10's letter**: "a human must approve the generated SQL before production apply." Keeping the apply in human hands makes the human checkpoint structural rather than procedural.
  3. **Cost-idle mode**: instances sleep by default; a CI migration step would need wake/sleep orchestration in the deploy path (mutating instances from CI — more IAM, more blast radius).
  4. **What we give up**: deploy-then-migrate ordering is not machine-enforced; a forgetful operator can deploy code before applying its migration. Mitigations: forward-only/additive migrations (§10) mean old-schema + new-code must tolerate each other within the rollback window anyway (ADR-012 mitigation note); the playbook orders the steps; the DB-asleep guard (D11) reminds at deploy time that migrations are operator-run.
- **Revisit when**: a real ops cadence exists (post-pilot) and Cloud Run jobs + Secret Manager-scoped short-lived DB creds can be designed properly (likely a `migrate` Cloud Run job under the backend SA, its own spec).

## D3. Image naming + immutability

- **Decision**: `europe-west1-docker.pkg.dev/tech-screen-493720/techscreen/{backend,frontend}:<full-40-hex-sha>-<env>`. One tag per push, no `latest`, no mutable convenience tags. `provenance: false` on buildx so each tag is a single image manifest, not an attestation list.
- **Rationale**: the full SHA + env makes the tag collision-free and lets the migration gate (D7) recover the deployed baseline by parsing the tag — the tag *is* the provenance link. Env in the tag is not decoration: frontend images differ per env by construction (`NEXT_PUBLIC_API_BASE_URL` is inlined at build — Dockerfile.frontend stage 4), and the backend carries it for symmetric audit even though its bytes are env-agnostic.
- **Alternatives considered**: short SHA (cloud-setup.md's older wording) — rejected: ambiguity risk is small but zero-cost to avoid; digest-pinning only — digests are what Cloud Run records anyway, but tags are the human/audit surface; **AR repository-level immutable tags** — the real enforcement, but it is a `google_artifact_registry_repository` attribute in `artifact_registry.tf`, outside this task's file scope; recorded as a follow-up. Until then immutability is by convention (nothing ever pushes the same sha-env twice except a byte-identical rebuild).

## D4. Deployer IAM: `run.developer`, not `run.admin` *(mandated research question)*

Role-by-role justification for `techscreen-deployer@` (all authored in `infra/terraform/iam.tf`, applied by the operator):

| Role | Scope | Why this and not more |
| --- | --- | --- |
| `roles/run.developer` | project | Grants `run.services.update` + get/list on services and revisions — enough to deploy new revisions **and** shift traffic (traffic is a field of the Service resource, so `update` covers `update-traffic`). What it deliberately lacks vs `run.admin`: `run.services.setIamPolicy` — the deployer can never widen public access (e.g. grant `run.invoker` to `allUsers` on something new) or re-bind invokers; that stays Terraform-owned. Project-level is acceptable: the project contains exactly the four TechScreen services (§8 v1.1 caps environments), and Cloud Run per-service IAM would buy little at the cost of four more bindings per future service. |
| `roles/iam.serviceAccountUser` | **on each of the four runtime SAs only** (SA-level `google_service_account_iam_member`) | Deploying a revision that runs as SA X requires `actAs` on X. SA-level bindings mean the deployer can act as exactly `techscreen-backend@`, `techscreen-backend-dev@`, `techscreen-frontend@`, `techscreen-frontend-dev@` — and **not** as `terraform@` (roles/owner) or `techscreen-flag-sync@`. A project-level `serviceAccountUser` would have handed the deployer a path to owner via the terraform SA; this is the single most important least-privilege line in the design. |
| `roles/artifactregistry.writer` | **repository `techscreen` only** (`google_artifact_registry_repository_iam_member`) | Push/pull on our one repo. Repo-level, not project-level: a future second repository (if ever) is not automatically writable. |
| `roles/cloudsql.viewer` | project | Read-only Cloud SQL metadata for the asleep guard (`sql instances describe` → `state`, `activationPolicy`). Chosen over `roles/cloudsql.client` deliberately: `client` carries `cloudsql.instances.connect` — a live connection path we do not want the deploy identity to possess even theoretically; `viewer` cannot touch data or connect. Broader than a custom role (it reads backups/databases metadata too) but custom roles are an ongoing maintenance tax the MVP does not need. |
| WIF binding | `roles/iam.workloadIdentityUser` for `principalSet://…/attribute.repository/kovalevgr/tech-screen-ai` | Same repository-pinned principalSet as `flag_sync` (iam.tf house pattern); the pool provider's attribute condition already pins the repo. No branch-level condition at MVP — the prod-ancestry check (D6) covers the "deploy random branch to prod" case at the workflow layer, and dev is *for* arbitrary refs. |
- **Alternatives considered**: `run.admin` (rejected above); one deployer SA **per environment** — rejected for MVP: the workflows are the authorisation boundary and both envs are same-trust (§8 single project); revisit if prod deploy rights ever diverge from dev. A dedicated `deployer-smoke@` identity for authenticated smoke — unnecessary: services are public (`allUsers` invoker), smoke is anonymous HTTP.

## D5. Reporting surface: job summary, not PR comment

- **Decision**: every workflow writes `$GITHUB_STEP_SUMMARY` (revision, image, URL, smoke verdict / traffic splits / rollback duration). No PR comments.
- **Rationale**: `workflow_dispatch` runs are not attached to a PR (the implementation plan's "PR comment" wording predates the T10 label design); the run page + summary is linkable, permanent, and shows the dispatching actor — the audit trail the playbook's `deploys` table will eventually formalise. The task input explicitly requests job summaries.

## D6. Prod ancestry guard

- **Decision**: for `env=prod`, the gate fails unless the resolved target SHA is an ancestor of `origin/main` (`git merge-base --is-ancestor`). `dev` accepts any ref.
- **Rationale**: the playbook's prerequisite "the PR is merged to main" becomes machine-checked for prod; dev is the rehearsal env (ADR-023) and must accept feature branches.

## D7. Migration-gate baseline + label verification **[repo-verified against T10's ci.yml]**

- **Decision**: baseline = git SHA parsed from the deployed backend image tag (`…/techscreen/backend:<sha>-<env>` → `<sha>`), validated with `git cat-file -e <sha>^{commit}`; fallback `origin/main~1` when the service runs the placeholder image or the SHA is not in history. Changed set = `git diff --name-only <baseline>..<target> -- alembic/versions/`. For each commit in `git log <baseline>..<target> -- alembic/versions/`, query `gh api repos/<repo>/commits/<sha>/pulls`; the commit passes iff ≥1 associated PR carries `migration-approved` (labels via `gh api repos/<repo>/pulls/<n>`). Any commit with zero associated PRs fails. Auth = the run's own `github.token` (`pull-requests: read`) — no extra secret.
- **Rationale**: diffing against *what is actually serving* is the §10-faithful question ("does this deploy introduce schema changes prod hasn't approved?"); the tag-parse works because D3 embeds the SHA. The `main~1` fallback matches the task brief and keeps the first-ever deploy from flagging the entire 0001–0005 chain (all long merged + already applied to both instances by T06). Honest limitation: on a placeholder baseline, migrations merged *before* the last PR are invisible to the gate — acceptable because application is operator-run anyway (D2) and the chain to date is applied; once one real deploy lands, the baseline is exact forever after.
- **Alternatives considered**: label check on the *current* PR only (T10's original sketch) — meaningless for `workflow_dispatch` (no PR context); a `deploys` DB table as baseline source — right long-term answer, out of scope (spec Out-of-scope); requiring a manual `confirmed_migrations: true` input — human-checkbox theatre that §10 explicitly upgraded from.

## D8. Traffic semantics: pin resolved revision names, never floating `LATEST`

- **Decision**: promote resolves `status.latestReadyRevisionName` and runs `update-traffic --to-revisions=<name>=<percent>`; the remainder redistributes proportionally across currently-serving revisions (gcloud's documented behaviour). Promote 100 pins `<name>=100` (not `--to-latest`). Deploy uses `--no-traffic`; the `candidate` tag moves via `update-traffic --update-tags candidate=<name>` (a tag move, no percent change). Rollback pins the detected previous revision at 100.
- **Rationale**: a floating `LATEST` allocation silently follows the *next* deploy — a later `/deploy` would instantly put the un-promoted candidate on live traffic percentage. Pinned names make every traffic state explicit and every shift a deliberate command; `--no-traffic` already converts any pre-existing LATEST assignment to a pinned one (first deploy after Terraform's default 100 %-to-LATEST state). `--update-tags` moves the tag off the previous candidate atomically — `deploy --tag` would collide with the existing assignment.
- **Edge encoded in the workflows**: promoting `<100` when only one revision serves traffic cannot distribute the remainder — gcloud errors; the summary explains. Promote of a revision already at the requested percent is a no-op success.

## D9. Rollback target resolution + measurement

- **Decision**: primary = revision with the highest percent in `status.traffic`; target = explicit `revision` input if given (readiness-verified), else the newest **ready** revision strictly older than the primary (walk `gcloud run revisions list --sort-by=~metadata.creationTimestamp`, skip until past primary, take the first with condition `Ready=True`). One `update-traffic --to-revisions=<target>=100`, wall-clock measured with `date +%s` around the call, duration in the summary with the ≤60 s target and the §19 5-minute ceiling stated.
- **Rationale**: "previous serving revision" must survive the case where the latest *created* revision is a failed deploy (skipped — not Ready) and the case of a mid-ramp rollback (primary at 90 %, candidate at 10 % → target is the primary's predecessor **only if** the primary is the candidate; walking creation order from the primary handles both because the primary is whatever holds the most traffic). The explicit input is the escape hatch when the heuristic is wrong at 02:00.
- **Concurrency**: all three workflows share group `cloud-run-<env>`; only rollback sets `cancel-in-progress: true` — rollback preempts, deploys/promotes queue.

## D10. Cost-idle mode: repo had no tooling — `scripts/cloud-sql-power.sh` ships here **[repo-verified gap]**

- **Finding**: the live instances are kept stopped (`activationPolicy=NEVER`) to save cost, but the tree contains **no** `scripts/cloud-sql-power.sh`, and `cloud-setup.md` has **no cost-idle section** — the practice exists only operationally. The guard message this task must emit references that script by path.
- **Decision**: ship the script in this PR (operator-only helper: `wake|sleep|status` × `dev|prod`, a thin `gcloud sql instances patch --activation-policy` wrapper) and document the rule in the deploy playbook. A one-line "steps that need the DB" table lives next to it.
- **Rationale**: the alternative — a deploy error message pointing at a file that does not exist — fails the operator at the worst moment. The script is 40 lines of shellcheck-clean bash, never run by CI (the deployer SA holds no `cloudsql.instances.update` — `viewer` cannot patch; waking is a human decision by design).

## D11. DB-asleep guard: conditional severity **[repo-verified against app/backend/main.py]**

- **Decision**: the gate describes the target instance (`state` + `settings.activationPolicy`); "awake" = `RUNNABLE` + `ALWAYS`. If asleep: **fail** iff backend is being deployed **and** the backend service template wires `DATABASE_URL` (detected via `gcloud run services describe` → container env names); otherwise **notice** and continue.
- **Rationale**: today's template wires nothing, `/health` is dependency-free, and the backend only opens the flag-service DB connection when `settings.database_url` is set (main.py lifespan) — so no deploy step touches the DB and hard-failing would block all deploys of a sleeping-DB world for no reason. The moment a follow-up wires `DATABASE_URL`, revision startup connects at boot → readiness needs the DB → the guard's fail branch activates *automatically*, because it reads the template rather than hardcoding an era. Steps that need the DB, documented: (1) backend revision startup **iff** `DATABASE_URL` is wired; (2) `alembic upgrade head` — operator-run, outside these workflows; (3) nothing else — build, push, traffic shifts, frontend smoke never touch it.

## D12. Known gap the first backend deploy will hit (not fixed here, loudly documented)

- **Finding** **[repo-verified]**: the `runtime` image bakes `APP_ENV=prod`; `Settings.llm_backend` defaults to `mock`; `assert_safe_for_environment()` raises on `prod`+`mock` at lifespan start. The T06 Cloud Run template sets **no env vars** (and no Cloud SQL attachment, no secret refs) — Terraform `ignore_changes` covers only image/client/scaling, so `/deploy` must not add env vars itself (they would be reverted-vs-drifted by the next `terraform plan`, breaking T06's SC-001 zero-diff contract).
- **Consequence**: the first *backend* `/deploy` builds, pushes, and then fails at the `gcloud run deploy` step (readiness probe: container exits on the RuntimeError). Frontend deploys are unaffected (Next.js has no such guard; `PORT` is honoured).
- **Decision**: T06a ships the correct mechanism and documents the failure mode (quickstart §5, playbook); the fix is a small follow-up in `infra/terraform/modules/environment/main.tf` — `env { LLM_BACKEND = "vertex" }` (+ `DATABASE_URL` secret ref + `cloudsql-instances` annotation when the backend gains DB routes) — outside this task's file boundary by explicit scope. Deliberately **not** smuggled in here: template changes are T06-contract territory and deserve their own reviewed plan diff.
- **Also noted**: `--port` (backend 8000 / frontend 3000) sets `template.containers[0].ports`; the provider treats `ports` as computed when unset in HCL, so no drift is expected — the quickstart schedules a `terraform plan` zero-diff check right after the first deploy to verify rather than assume.
