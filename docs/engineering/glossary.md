# Glossary

Domain terms used throughout TechScreen. When you encounter one of these terms in code, a prompt, an ADR, or a spec, it means the exact thing defined here. If a concept needs a different meaning in a specific context, we pick a different term rather than overload.

Listed alphabetically within sections.

---

## Core concepts

**Assessor (agent).** The runtime LLM agent that evaluates a candidate's turn and produces a structured JSON assessment per competency. Runs once per candidate turn. Output goes into the `assessment` table and is never mutated (see **append-only audit trail**).

**Audit log.** The `audit_log` table. Records every state change with actor, action, subject, timestamp. Append-only.

**Calibration.** Measuring how closely the Assessor's automated scores agree with human reviewer scores. Outputs agreement metrics (exact match, within-0.5, systematic bias per competency). Runs in CI as a warning-only check. See `./testing-strategy.md` and the `calibration-run` skill.

**Candidate.** The person being interviewed. Identified by a row in the `candidate` table. Authenticates via magic link — never holds an account (see ADR-016).

**Competency.** A named skill area evaluated in an interview (e.g. "C# concurrency", "React state management", "system design"). A rubric node with `kind: competency`. Competencies have sub-competencies and are scored on a 1–5 level scale.

**Correctness (Variant A / B / C).** The three approaches we considered for flagging factually wrong answers. The MVP uses Variant A (prompt the Assessor itself). See ADR-020.

**Depth-probe branch.** A pre-planned follow-up question in the `InterviewPlan` that triggers when the candidate's first answer is judged shallow. Allows the Interviewer to dig deeper without ad-lib reasoning.

**Extended plan.** A future (Roadmap H1) version of the `InterviewPlan` that includes branching logic beyond simple depth-probes.

**Interviewer (agent).** The runtime LLM agent that conducts the dialogue with the candidate. Runs once per candidate-facing turn. Uses the **InterviewPlan snapshot** as a scaffold.

**InterviewPlan.** The structured plan produced by the **Pre-Interview Planner agent** before a session. Contains seed questions per competency, depth-probe branches, opening/closing scripts, time budget. Recruiter-reviewable; frozen as `interview_plan_snapshot` at session start.

**Interview session.** A single conducted interview. Corresponds to one `interview_session` row. Holds references to the frozen rubric snapshot, plan snapshot, and assigned reviewer.

**Level (1–5).** The score a competency receives. 1 = "Superficial", 5 = "Expert". Levels are tied to anchors in the rubric and have behavioural descriptions per level per competency.

**Magic link / magic token.** The candidate's authentication mechanism. A signed, single-use, short-lived token delivered by email, redeemed to enter the interview UI. See ADR-016.

**Position template.** A configuration (`configs/positions/*.yaml`) describing a hiring role: seniority, tech stack, competencies in scope, time budget. Recruiters pick a position template when creating a session.

**Pre-Interview Planner (agent).** The offline LLM agent that produces an `InterviewPlan` from the position template and current rubric. Runs once per session, before the interview starts. Uses Gemini 2.5 Pro (see ADR-003).

**Recruiter.** An internal N-iX user who creates interview sessions, reviews InterviewPlans before the interview, and sees the final report. Authenticates via Google Workspace SSO.

**Reviewer.** An internal N-iX user who audits completed sessions. Can override Assessor decisions via `assessment_correction`. Overlaps with the recruiter role but is a distinct permission.

**Rubric.** The tree of competencies, sub-competencies, and level anchors used by the Assessor. Versioned via `rubric_tree_version`.

**Rubric node.** A node in the rubric tree. Has a `kind` enum: `root`, `group`, `competency`, `sub_competency`, `anchor`. Has a `stable_id` that persists across renames for longitudinal analysis.

**Rubric snapshot.** The resolved rubric tree frozen into `interview_session.rubric_snapshot` at session start. The Assessor evaluates against the snapshot, not the live rubric (see ADR-018).

**Seed question.** A rubric-anchored starting question for a competency. Generated by the Pre-Interview Planner, stored in the InterviewPlan, surfaced by the Interviewer.

**Session decision.** The final hiring recommendation for a candidate after an interview: `hire`, `no_hire`, `borderline`. Lives in `session_decision` table. Append-only; revised via a new row (see ADR-019).

**Turn.** One exchange in the interview — a candidate message plus the immediately preceding interviewer prompt it responds to. One row in the `turn` table per candidate message.

**Turn annotation.** A reviewer-added quality mark on a specific turn (e.g. "off-topic", "hallucinated reference", "strong answer"). Lives in `turn_annotation` — append-only.

**Turn trace.** The full record of every LLM call made while processing a turn: model, prompt inputs, raw output, latency, cost, token counts. Lives in `turn_trace`. Append-only. Used for debugging, cost analysis, and calibration replay.

## Infrastructure & process

**Append-only audit trail.** The principle that certain tables are never updated or deleted by application code. See constitution §3 and ADR-019. Affects: `turn_trace`, `assessment`, `assessment_correction`, `turn_annotation`, `audit_log`, `session_decision`.

**Bootstrap.** The one-time `infra/bootstrap.sh` execution that creates the GCS state bucket, Terraform service account, and Workload Identity Federation pool. Runs manually once per GCP project; everything after runs via Terraform.

**Configs as code.** The principle that rubrics, position templates, prompts, and feature flag defaults live in Git. Admin UI edits are promoted via PR. See ADR-021.

**Dark launch.** Shipping a feature behind a feature flag that starts `false`. Flag is flipped on after smoke and calibration checks pass in production. See constitution §9 and ADR-011.

**Deploy checkpoint.** The human approval step between a Cloud Run revision at 0% traffic and first user-facing traffic at 10%. Gated on green smoke tests and reviewed migration SQL.

**Docker parity.** The principle that dev, CI, and prod all run the same multi-stage Docker images. See ADR-010.

**Feature flag.** A boolean toggle in the `feature_flag` table (self-hosted, see ADR-011) that gates a code path. Defaults live in `configs/feature-flags.yaml`.

**Forward-only migration.** A migration that adds, never removes. Column drops require a multi-step migration: add new, dual-write, backfill, remove reads, then drop.

**Plan (Spec Kit).** The `.specify/specs/<slug>/plan.md` produced by the `/plan` command. Contains task-level agent assignments and parallel groupings. See ADR-014.

**Prod-only topology.** The principle that production is the only long-lived environment. Pre-prod verification happens via Cloud Run revisions at 0% traffic. See ADR-009.

**Promote.** The `/promote N` slash command that shifts Cloud Run traffic percentage (10 → 50 → 100) to the new revision. Human-initiated on MVP.

**Rollback.** The `/rollback` slash command that shifts Cloud Run traffic back to the previous stable revision. Completes in under a minute. See constitution §19.

**Smoke test.** A minimal test suite executed against a new Cloud Run revision receiving 0% traffic, before any user sees it. Blocks `/promote`.

**Spec (Spec Kit).** The `.specify/specs/<slug>/spec.md` produced by `/specify`. Natural-language description of what changes and why.

**Traffic splitting.** Cloud Run's native capability of dividing incoming requests across multiple revisions by percentage. Our canary mechanism. See ADR-012.

**Workload Identity Federation (WIF).** OIDC-based authentication between GitHub Actions and GCP. Short-lived tokens, no JSON service-account keys. See ADR-013.

## Agent / AI concepts

**Agent contract.** The documented input/output schema of an LLM agent, including prompt structure, expected JSON fields, and invariants. Lives in `../specs/agents.docx` and `prompts/<agent>/<version>/`.

**Context window.** The maximum number of tokens a model can process in a single call. We budget session prompts to stay well below model limits with headroom for growth.

**Hybrid language.** The convention that agent system prompts are in English while candidate-facing output is in Ukrainian. See ADR-008.

**Level prompting guide.** A section in agent prompts that specifies how to calibrate tone and question depth across seniority levels (entry / specialist / expert / proficient).

**Multi-agent fan-out.** Dispatching tasks to sub-agents (backend-engineer, frontend-engineer, etc.) to run in parallel. Explicit, not automatic (see ADR-014).

**Orchestrator.** The deterministic Python state machine that drives an interview. Dispatches to LLM agents but makes all flow decisions itself. See ADR-005.

**Sub-agent (Claude Code).** A specialised assistant defined under `.claude/agents/<name>.md` with a focused system prompt and tool allow-list. Used for multi-agent development work — not the same as the runtime Interviewer/Assessor/Planner agents.

**Ukrainian style anchors.** A block of Ukrainian exemplars attached to agent system prompts that constrain register, phrasing, and tone in candidate-facing output.

## Diagrams, versioning, reviews

**ADR (Architecture Decision Record).** A short document capturing one non-trivial decision in Michael Nygard format (Context / Decision / Consequences). See `adr/README.md`.

**Constitution.** `.specify/memory/constitution.md` — the 20 non-negotiable invariants of the project.

**Design token.** A named design value (colour, spacing, typography scale) that the UI references instead of hardcoded values. Lives in `docs/design/tokens/`.

**Reviewer agent (Claude Code).** A read-only sub-agent that validates constitution adherence, secrets scan, test coverage, and migration safety. Runs on every PR.

**Spec Kit.** [GitHub Spec Kit](https://github.com/github/spec-kit). Our spec-driven development workflow. See ADR-017.

**Stable ID (rubric).** A long-lived identifier on a rubric node that survives node rename or reorganisation. Used for longitudinal queries across rubric versions.
